# ТЗ №1 — Sotto (локальный Whisper + опциональная облачная пост-обработка)


# ТЗ: Sotto — локальный клон Whisper Flow для macOS

**Название приложения:** **Sotto** (от итал. *sotto voce* — «вполголоса»).
Используется в config-пути (`~/.config/sotto/`), bundle id и LaunchAgent.

## Цель
Нативная по ощущениям утилита диктовки для macOS (Apple Silicon): я зажимаю
ПРАВУЮ Option в любом приложении, говорю, отпускаю — распознанный текст
вставляется в активное поле ввода, как будто я его напечатал. Референс по UX —
Wispr Flow (wisprflow.ai): изучи их сайт, чтобы понять продукт (push-to-talk,
вставка в любое приложение, авто-пунктуация, словарь пользователя), но код
пиши с нуля.

## Ключевые требования
1. **Push-to-talk**: зажатие клавиши-триггера начинает запись с микрофона,
   отпускание — останавливает. По умолчанию триггер — **правая Option**
   (`Key.alt_r`), при этом левая Option работает как обычно. Клавиша-триггер
   **настраивается** через конфиг (`hotkey` / `trigger_key`) — можно назначить
   другую клавишу или комбинацию; правый Option задан как разумный дефолт.
   Индикация записи обязательна (иконка в menu bar меняется + опционально
   маленький floating-оверлей с уровнем звука).
2. **Распознавание — локально, бесплатно, офлайн**:
   - основной бэкенд: mlx-whisper (оптимизирован под Apple Silicon) либо
     whisper.cpp через pywhispercpp — выбери сам и обоснуй в плане;
   - модель по умолчанию `large-v3-turbo` (сравни с `medium`
     по скорости/качеству на Apple Silicon и выбери дефолт), язык — авто
     (я диктую на русском и английском вперемешку);
   - модель скачивается при первом запуске, дальше полный офлайн.
3. **Вставка текста**: через буфер обмена (pbcopy → эмуляция Cmd+V), с
   сохранением и восстановлением прежнего содержимого буфера. Fallback —
   поимпульсный ввод через CGEvent, если вставка недоступна.
4. **Пост-обработка (опционально, выключена по умолчанию)**: конфиг-флаг
   `postprocess.enabled`. Если включён — сырой транскрипт прогоняется через
   LLM по OpenAI-совместимому API (base_url + api_key + model в конфиге; я
   буду использовать OpenRouter). Задача LLM: пунктуация, капитализация,
   исправление доменных терминов по пользовательскому словарю
   (`vocabulary: ["Claude Code", "Fable", "MCP", ...]` в конфиге). Ключ —
   только из конфига/env, никогда не хардкодить. При недоступности API —
   молча вставлять сырой текст.
5. **Menu bar app**: иконка со статусом (idle / recording / transcribing),
   меню: включить/выключить, выбор модели, выбор микрофона, переключатель
   пост-обработки, последние 5 транскриптов (клик — скопировать), выход.
6. **Конфиг**: один YAML/TOML в `~/.config/<имя-приложения>/`, создаётся с
   дефолтами при первом запуске.
7. **Автозапуск**: LaunchAgent + команды install/uninstall.

## Технические рамки
- Язык: Python 3.11+ (venv, зависимости через requirements.txt или uv) ИЛИ
  Swift — выбери сам, но обоснуй выбор в плане с учётом надёжности глобального
  перехвата клавиш и простоты сопровождения. Не Electron.
- macOS-разрешения (Microphone, Accessibility/Input Monitoring): при первом
  запуске проверять и выводить понятную инструкцию, какие галочки поставить в
  System Settings; не падать молча.
- Латентность цель: ≤ 2 сек от отпускания клавиши до вставки текста на фразе
  10–15 секунд (без пост-обработки).
- Запись: 16 kHz mono WAV во временный файл в памяти/tmp, удалять после
  распознавания. Ничего не логировать из содержимого диктовок.

## Порядок работы
1. Сначала напиши краткий план: выбранный стек, архитектура (модули), риски
   (права macOS, перехват Option, конфликт с раскладками) и как их решаешь.
   Потом сразу реализуй — подтверждения не жди.
2. Реализуй проект целиком: код, README (установка, права, настройка,
   troubleshooting), Makefile или скрипт `run.sh` / `install.sh`.
3. Проверь всё, что можно проверить без микрофона: юнит-тест на цикл
   «WAV-файл → транскрипт» с заранее сгенерированным тестовым аудио
   (сгенерируй через `say -o test.aiff` + конвертация), тест сохранения/
   восстановления буфера, синтаксис/линт.
4. В конце выведи: команду установки, команду запуска и чек-лист ручной
   проверки (5 пунктов).

Работай автономно до конца, не задавай уточняющих вопросов — при
неоднозначности принимай разумное решение и фиксируй его в README в разделе
"Design decisions".

## Примечания

- «Realtime»-вставка в видео на самом деле не показана: текст появляется после отпускания клавиши. Так и заложено в v1. Streaming-распознавание по мере речи — отдельная доработка.
- OpenRouter не даёт нормального speech-to-text — поэтому распознавание локальное, а облако (OpenRouter) используется только для опциональной LLM-пост-обработки (пунктуация, словарь терминов). Если захочется облачный STT — это OpenAI-совместимый endpoint `/audio/transcriptions` (OpenAI, Groq), можно добавить как альтернативный бэкенд.
